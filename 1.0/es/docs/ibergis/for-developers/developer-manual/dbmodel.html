Esta sección describe la estructura y organización del modelo de base de datos utilizado por IberGIS.

2.1. Formato de Archivo de Datos

IberGIS utiliza archivos GeoPackage (GPKG) como formato principal de almacenamiento de datos. GeoPackage es un formato abierto, basado en estándares, independiente de plataforma, portátil, autodescriptivo y compacto para transferir información geoespacial. Está basado en el motor de base de datos SQLite 3 y soporta tanto características vectoriales como datos ráster.

Los archivos de proyecto principales creados por IberGIS tienen la extensión .gpkg y contienen todas las tablas necesarias, datos espaciales, configuraciones y metadatos para proyectos de modelado hidráulico.

2.2. Organización del Esquema de Base de Datos

El esquema de base de datos está organizado en el directorio dbmodel/ dentro de la carpeta de instalación del plugin. La estructura sigue un enfoque modular, separando diferentes aspectos de la base de datos en componentes lógicos.

2.2.1. Estructura de Archivos SQL

El modelo de base de datos se define mediante múltiples archivos SQL organizados en subdirectorios.

2.2.1.1. DDL (Lenguaje de Definición de Datos)

Ubicado en dbmodel/ddl/ddl.sql.

Este archivo contiene todas las definiciones de tablas, incluyendo:

  • Tablas de configuración: Almacenan parámetros del plugin y configuraciones de usuario.

  • Tablas de catálogo: Definen tipos y categorías disponibles para varios elementos (escenarios de contorno, materiales, curvas, etc.).

  • Tablas de entidades: Almacenan los principales elementos hidráulicos y geométricos.

  • Tablas de sistema: Gestionan operaciones internas del plugin y configuraciones de formularios.

  • Tablas de geometría: Almacenan entidades espaciales para terrenos, techos, entradas, conductos, nodos y otros elementos.

Todas las tablas incluyen restricciones de verificación de tipo de SQLite para garantizar la integridad de datos.

2.2.1.2. DML (Lenguaje de Manipulación de Datos)

Ubicado en dbmodel/dml/dml.sql.

Este archivo contiene las sentencias de población de datos inicial:

  • Configuración del sistema de referencia espacial (SRID): Configura el sistema de referencia de coordenadas.

  • Inicialización de tablas de sistema: Puebla las tablas de configuración y metadatos.

  • Datos de catálogo: Inserta valores predeterminados para todas las tablas de catálogo.

  • Definiciones de campos de formulario: Configura los formularios dinámicos utilizados en todo el plugin.

  • Valores de tipo: Puebla listas desplegables y opciones seleccionables.

El archivo DML usa marcadores de posición como <SRID_VALUE> que se reemplazan dinámicamente al crear nuevos proyectos.

2.2.1.3. Disparadores

Ubicado en dbmodel/trg/.

  • trg_create.sql: Contiene disparadores CREATE que:

    • Generan automáticamente códigos únicos para elementos nuevos (por ejemplo, “J1”, “C5”, “RF10”).

    • Mantienen relaciones topológicas entre nodos y arcos.

    • Actualizan relaciones de tabla padre-hijo.

    • Sincronizan cambios de geometría en tablas relacionadas.

  • trg_delete.sql: Contiene disparadores DELETE que:

    • Propagan eliminaciones en cascada a tablas relacionadas.

    • Limpian registros huérfanos.

    • Mantienen la integridad referencial.

2.2.1.4. Funciones

Ubicado en dbmodel/fct/.

  • fct_after_import_ground_geometries.sql: Post-procesamiento para importaciones de polígonos de terreno.

  • fct_after_import_inlet_geometries.sql: Post-procesamiento para importaciones de puntos de entrada.

  • fct_after_import_roof_geometries.sql: Post-procesamiento para importaciones de polígonos de techo.

  • fct_after_import_inp.sql: Post-procesamiento después de importar archivos EPA SWMM .inp.

Estas funciones manejan la normalización de datos, asignaciones de código y establecimiento de relaciones después de importaciones masivas.

2.2.1.5. Internacionalización (i18n)

Ubicado en dbmodel/i18n/.

Contiene archivos DML específicos de idioma para múltiples localizaciones:

  • en_US/dml.sql: Inglés (Estados Unidos).

  • es_ES/dml.sql: Español (España).

  • es_CR/dml.sql: Español (Costa Rica).

Estos archivos proporcionan valores traducidos para cadenas de cara al usuario almacenadas en la base de datos, como etiquetas de campos, información sobre herramientas y mensajes.

2.2.1.6. Base de Datos de Informes

Ubicado en dbmodel/rpt_gpkg/.

Define un esquema separado para almacenar resultados de simulación:

  • ddl.sql: Tablas para almacenar resultados de simulación EPA SWMM (nodos, arcos, resúmenes de flujo, etc.).

  • dml.sql: Configuración inicial para tablas de resultados.

La estructura de base de datos de informes permite a IberGIS importar y visualizar salidas de simulación directamente en QGIS.

2.2.1.7. Tablas de Sistema

Ubicado en dbmodel/sys_gpkg/sys_gpkg.sql.

Crea tablas de nivel de sistema requeridas por GeoPackage y GDAL/OGR:

  • gpkg_ogr_contents: Rastrea recuentos de entidades para cada tabla.

  • Tablas de metadatos adicionales para cumplimiento de GeoPackage.

2.2.1.8. Datos de Ejemplo

Ubicado en dbmodel/example/example_data.sql.

Proporciona datos de muestra para propósitos de prueba y demostración, incluyendo:

  • Condiciones de contorno con geometrías.

  • Entidades de muestra de terreno, techo y entrada.

  • Escenarios preconfigurados.

  • Hietogramas y series temporales de ejemplo.

2.3. Inicialización de Base de Datos

El archivo dbmodel/init.sql sirve como punto de entrada para la creación de la base de datos. Hace referencia y ejecuta todos los demás archivos SQL en el orden correcto para construir una base de datos de proyecto completa.

La secuencia de inicialización típica es:

  1. Crear tablas de sistema GeoPackage.

  2. Ejecutar DDL para crear todas las tablas.

  3. Ejecutar DML para poblar datos iniciales.

  4. Aplicar internacionalización para el idioma seleccionado.

  5. Crear disparadores para integridad de datos.

  6. Cargar datos de ejemplo (si se crea un proyecto de muestra).

2.4. Control de Versiones y Actualizaciones

Ubicado en dbmodel/updates/.

El plugin incluye un sistema de versionado organizado por números mayor/menor/parche:

  • updates/1/1/0/: Contiene scripts de actualización para la versión 1.1.0.

    • changelog.txt: Describe los cambios en esta versión.

    • ddl.sql: Modificaciones de esquema.

    • dml.sql: Migraciones de datos.

Esta estructura permite a IberGIS actualizar automáticamente las bases de datos de proyectos existentes cuando los usuarios actualizan el plugin a una versión más reciente.

Importante

Las actualizaciones también deben aplicarse a los archivos base para proyectos nuevos. Por ejemplo, si la versión 1.1.0 introduce una nueva tabla, debe actualizar dbmodel/ddl/ddl.sql para incluir la nueva definición de tabla además de actualizar dbmodel/updates/1/1/0/ddl.sql.

2.5. Características Clave

2.5.1. Verificación de Tipo

Todas las tablas usan la función typeof() de SQLite en restricciones CHECK para aplicar validación de tipo estricta, previniendo corrupción de datos por tipos de valor incorrectos.

2.5.2. Datos Espaciales

Las entidades geométricas usan el tipo de geometría GeoPackage y se registran en la tabla estándar gpkg_geometry_columns. Los tipos de geometría admitidos incluyen:

  • POINT (entradas, uniones, almacenamiento, desagües, divisores, hietogramas).

  • LINESTRING (conductos, orificios, vertederos, bombas, salidas).

  • POLYGON (terrenos, techos).

  • MULTILINESTRING (condiciones de contorno).

2.5.3. Claves Foráneas

La base de datos usa extensivamente restricciones de clave foránea para mantener la integridad referencial entre tablas relacionadas. Las actualizaciones en cascada y las eliminaciones restringidas aseguran la consistencia de datos.

2.5.4. Campos JSON

Algunas tablas de configuración usan campos TEXT formateados JSON para almacenar datos estructurados complejos, como parámetros de widgets de formulario y metadatos adicionales.
